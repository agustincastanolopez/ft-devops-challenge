The Scenario

A new microservice called event-enricher needs to go to production. It:

  • Consumes raw player events from an MSK Kafka topic (player.events.raw)
  • Enriches each event by reading player profile metadata from Aurora MySQL RDS
  • Deduplicates events within a 60-second window using ElastiCache (Redis)
  • Publishes enriched events to a second Kafka topic (player.events.enriched)
  • Enqueues high-priority events (deposit, churn_risk) onto a RabbitMQ queue for downstream CRM activity triggers
  • Writes a denormalized event record to ClickHouse for analytics consumption
  • Exposes /healthz and /readyz endpoints for EKS probes

Stub Go implementations of all consumers and producers are provided. You do not need to build the business logic — only the infrastructure and operational layer around it.

Important: You are not expected to deploy to a live AWS account. All infrastructure (Terraform, K8s manifests, CI/CD) should be validated locally — e.g., terraform validate/plan, kubectl apply --dry-run=client, docker build, go test ./.... The goal is production-ready artifacts, not a running environment.

Constraints

AWS region eu-west-1
Environments staging and production namespaces minimum

Security See Security Requirements below
Multi-tenancy Events carry an operator_id — observability must be operator-aware

Security Requirements

Area Expectation
Data protection No PII (e.g., player_id, email, IP) in logs, metrics labels, or Kafka message keys — treat player data as sensitive
Secrets management No hardcoded credentials anywhere — use IRSA for AWS services, K8s Secrets or External Secrets Operator for connection strings
Network security Least-privilege security group rules — the service should only reach the services it needs
Container security Non-root runtime, minimal base image, no shell in production containers
IAM Least-privilege IAM policies scoped to exactly the resources the service needs
Supply chain Consider dependency scanning and/or image scanning in your CI/CD pipeline
Encryption in transit TLS for all service-to-service communication (MSK, RDS, Redis, RabbitMQ all support it)

Testing

Starter tests are provided in main_test.go and enricher/enricher_test.go covering health endpoints, configuration loading, and pipeline processing. Your CI/CD pipeline should run go test ./... and golangci-lint.

You are expected to extend the test suite as you build out the service — for example, testing your observability instrumentation, configuration edge cases, or any logic you add. The existing tests are a baseline, not the finish line.
